# 标准观测路径之外：一次基于 Netlink/Devlink 的 NIC 静默故障定位实践
> ——从标准日志失效，到通过 Netlink/Devlink 主动观测 NIC 固件瞬态状态异常

## 前言
在底层系统故障排查中，最麻烦的一类问题，并不是系统直接崩溃。
而是：
- 业务已经出现异常
- 但系统没有明确报错
- 日志体系未观测到有效错误信息

这种问题的难点不在于技术复杂度本身，而在于：
> 当驱动与内核层级的标准观测路径未提供有效信息时，如何重新建立对底层运行状态的观测能力。

本文记录一次 ESD（静电放电）测试中的NIC故障排查过程，并基于该案例讨论：
- 为什么“未观测到错误日志”不等于底层系统没有异常
- 为什么驱动侧 event-driven 上报机制存在天然观测盲区
- 如何从被动等待错误上报，转向主动采集底层运行状态
- 如何依托原生内核通信能力，补齐底层诊断观测缺口

> 注：因涉及生产环境，相关代码不便公开，本文重点分享排查思路与通用方法论。

## 一、问题背景：业务流量中断，但系统看起来正常
问题最初出现在服务器 ESD 测试过程中，在受到静电干扰后，NIC出现偶发性业务流量中断。

该现象具有一个明显特征：
**在高负载条件下，触发概率更高。**

但在常规运行及性能测试条件下，NIC业务流量表现正常。

## 二、标准观测路径未提供有效信息
故障出现后，首先使用 Linux 常规底层观测路径进行排查，包括：
- 业务层NIC日志
- 内核 dmesg 日志
- PCIe 设备在线状态
- 驱动模块加载状态

排查结果如下：
- 业务层日志报出基于收发包监控发现的异常
- 内核日志未看到错误或告警信息
- PCIe device 始终处于在线状态
- driver module 正常加载

在当前观测范围内，该问题表现出典型的底层静默故障特征：底层异常已经发生，但标准观测路径没有提供有效异常信息。

## 三、驱动事件上报机制的观测边界
进一步分析后，问题可以收敛为两种可能：
- 驱动感知到底层异常，但未触发上报机制
- 驱动未识别到底层异常

无论哪种情况，本质都指向同一个问题：
**基于 event-driven 的驱动可观测机制存在固有观测边界** 。

标准驱动的健康上报与日志机制，其设计目标通常是：
- 降噪
- 结构化输出
- 聚焦关键错误事件

而非完整记录所有底层状态变化。

因此，以下类型的状态变化可能不会进入标准观测路径：
- 持续时间极短、可自动恢复的瞬态状态波动
- 未达到错误阈值的状态偏移
- 不满足预定义错误模型的固件状态切换

这些状态在当前系统中可能不会同步进入：
- dmesg
- kernel event log
- 上层监控系统

这就形成了底层排查中的核心误区：
> 日志无报错 ≠ NIC底层运行无异常

本次 ESD 引发的流量中断，恰好超出了这一观测边界。

## 四、转向主动观测：Netlink + Devlink
当标准观测路径无法提供有效信息后，排查策略需要切换。

问题不再是：
> 如何分析已有日志

而是：
> 如何重新建立对底层运行状态的观测能力

标准观测路径（dmesg/ethtool/sysfs）依赖驱动预定义上报逻辑，一旦异常未被触发，观测链路可能中断；而 Devlink health 提供了另一种能力：
> 通过驱动暴露的接口，对固件健康状态进行主动查询。

在本次案例中，**基于 Netlink 的 Devlink health interface**，构建了一套 monitoring 模块，用于定期采集NIC固件运行状态。

该方案特点如下：
- 基于标准内核通信机制，支持用户态与内核态交互
- 不依赖驱动事件触发逻辑
- 支持对NIC固件状态进行周期性采样
- 适用于捕获短时、高负载相关的瞬态异常

这里的核心价值并不是替代驱动现有日志与 event-driven 上报机制，而是：
> 在标准观测路径之外，补充一种额外的底层运行状态观测能力。

## 五、异常定位：标准观测体系外捕获到状态波动
在 ESD 叠加高负载的测试窗口内，通过 monitoring 模块，观测到NIC固件健康状态出现了短时波动。

该波动与业务流量中断在时间上呈现高度相关性。

同时观测到：
- 状态波动持续时间极短
- 状态可恢复
- 未触发驱动预设错误阈值

由于未满足标准错误模型触发条件，该类异常未进入：
- 内核事件系统
- 驱动日志系统
- 标准监控系统

因此，在系统层面表现为：
- 业务流量中断
- 日志体系未观测到异常
- 驱动状态显示正常

最终在当前观测条件下确认：
> 异常行为确实存在，但未被标准观测体系覆盖。

## 六、本次案例的核心工程启示
这个案例最大的价值，在于直观体现了：**标准驱动观测路径的边界**。

event-driven 上报机制，通常更适合处理：
- 持续性硬件故障
- 稳定可复现异常
- 达到预设触发条件的错误事件

但对于以下场景，在部分实现中可能存在观测限制：
- 高负载 + 电气扰动引发的短时波动
- 快速自愈、无持续报错的底层异常
- 未触发标准事件上报机制的状态变化

因此：
> 日志沉默本身，也是一种信号。

它往往意味着：**当前观测体系未覆盖实际发生的异常行为**。

## 七、沉淀：日志沉默型底层故障的通用排查思路
基于本次案例，最终沉淀出一套适用于驱动 / 固件层级静默故障的通用排查路径：

### 1. 不默认现有观测体系完整性
内核日志、驱动事件、常规工具输出，本质上都是：
“**系统当前暴露的状态**”，而非完整系统状态。

### 2. 区分“系统正常”与“未观测到异常”
大量短时 、自愈类底层异常，不会触发标准告警。
因此：
> 未观测到异常 ≠ 系统无异常。

### 3. 被动观测失效后，及时切换主动观测
当常规被动观测路径无法提供有效信息时，应引入**主动采集状态机制**。

### 4. 向下穿透层级，聚焦驱动/固件边界
多数无日志静默故障，其问题往往出现在：
- driver runtime state
- firmware transient state
- 软硬件交互边界

## 结语
底层系统排查最难的，从来不是复杂报错。
而是：
> 在标准观测路径中未被记录的异常。

这次排查让我意识到：诊断能力的本质不只是分析日志，而是**在观测链路不完整时，重新构建对系统状态的可见性**。
